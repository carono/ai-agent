# Журнал ручных вмешательств

Этот документ — контракт формата журнала, в который агенты записывают факты, когда пользователь поправил их работу руками. Журнал — сырой поток событий; фильтры (например «трение», т.е. повторяемые правки) и аналитика — отдельные задачи поверх этого потока.

---

## Где лежит журнал

```
workflow/{bot-name}/manual-interventions.jsonl
```

Один файл на бота, формат — JSON Lines (одна запись = одна строка JSON, без массива и без обёртки). Файл коммитится вместе с остальной конфигурацией бота, если бот в режиме `project`; в режиме `local` — попадает под общий `.gitignore` папки бота.

Если файла ещё нет — агент создаёт его при первой записи.

---

## Что считается вмешательством

В журнал попадают **только** факты, которые агент в рамках своей сессии может **точно идентифицировать** как правку **своей** работы:

1. Пользователь в текущей сессии явно сказал «был косяк», «поправил руками», «не так сделал» или аналогично.
2. В комментарии к код-ревью прямо указано на ошибку агента (а не «направление развития» или новое требование).

**Не считается вмешательством и в журнал не пишется:**

- Любые правки кода другим разработчиком, не связанные с сессией бота. Если агент не может с уверенностью сказать «это правили мою работу» — записи нет.
- Замечания к MR, которые формулируют новое требование или меняют постановку задачи (это работа `discussion`, не правка).
- Изменения пользователем `WORKFLOW.md` / `bot.local.md` / других конфигов воркфлоу — это настройка, не вмешательство.
- Закрытие задачи без мёржа («передумали») — это решение по продукту, не правка работы агента.

Принцип: лучше не записать спорный случай, чем зашумить журнал.

---

## Поля записи

Минимальный набор, известный агенту из воркфлоу трекера. Виноватых не ищем, severity и commit-sha не пишем.

| Поле | Тип | Значение |
|---|---|---|
| `timestamp` | string (ISO 8601, UTC) | Момент фиксации, например `2026-04-27T10:15:30Z` |
| `agent` | string | Какой агент работал: `discussion`, `worker`, `reviewer` |
| `task` | string | Идентификатор задачи в трекере, например `WF-32` |
| `step` | string | Этап воркфлоу: `discussion`, `development`, `review` |
| `what` | string | Краткое описание факта вмешательства одной строкой |

Все поля обязательны. Лишние поля не добавляются.

---

## Примеры записей

```jsonl
{"timestamp":"2026-04-27T10:15:30Z","agent":"worker","task":"WF-32","step":"development","what":"пользователь руками поправил формат таблицы полей в MANUAL_INTERVENTIONS.md"}
{"timestamp":"2026-04-27T14:02:11Z","agent":"reviewer","task":"WF-21","step":"review","what":"в комментарии MR указано: ссылка на задачу собрана вручную из UUID, надо брать из get-task.url"}
```

---

## Как агент пишет в журнал

Агент дописывает строку в конец файла в момент, когда вмешательство явно идентифицировано — в той же сессии, где это произошло. Хуки и внешние скрипты в этой задаче не предусмотрены: запись делает сам агент.

Пример команды (одна строка JSON, аппенд):

```bash
printf '%s\n' '{"timestamp":"2026-04-27T10:15:30Z","agent":"worker","task":"WF-32","step":"development","what":"..."}' \
  >> workflow/{bot-name}/manual-interventions.jsonl
```

Если агент не может сформулировать факт в одну строку без гадания — запись не делается.
