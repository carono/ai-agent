## Шаг 0.7 — Принцип автономии

Главная установка: **по умолчанию принимай решение сам**, вопрос задавай только если цена ошибки высокая. Чем больше ты работаешь с проектом, тем меньше должно быть вопросов — накопленный контекст в `PROJECT.md` и `knowledge/` должен закрывать всё больше неясностей.

**Вопрос обязателен**, если выполняется хотя бы одно:
- Решение необратимо или дорого откатить (схема БД, миграции, публичный API-контракт, удаление/перезапись данных)
- Затрагивает несколько модулей или внешние интеграции, и нет однозначного выбора в коде/документах
- Бизнес-правило не выводится из `PROJECT.md`, `knowledge/`, кода и закрытых задач
- Между путями реализации стоимость отличается значительно (часы и более) и выбор зависит от приоритетов, которых нет в документах

Во всех остальных случаях — фиксируй интерпретацию в итоговом summary («понял так: X. Если не так — верните в обсуждение, иначе беру в работу») и переводи в «готово к разработке».

**Перед любым вопросом** — попробуй найти ответ в:
1. Описании задачи и комментариях
2. `workflow/{bot-name}/PROJECT.md` и `bot.local.md`
3. `workflow/{bot-name}/knowledge/` — если есть `INDEX.md`, читай только релевантные темам файлы; всегда смотри `decisions.md` (если есть) — там прошлые принятые решения
4. Коде проекта (Read, Grep, Glob), включая похожие закрытые задачи

Если ответ выводится — не спрашивай. Не задавай ритуальные вопросы (бизнес-цель, критерии приёмки, тех.детали) когда они уже понятны из контекста.

## Шаг 1 — Изучи контекст проекта

Прочитай:
- `workflow/{bot-name}/WORKFLOW.md` — как взаимодействовать с трекером, статусы, API
- `workflow/{bot-name}/PROJECT.md` — бизнес-контекст, модули, интеграции, термины
- `workflow/{bot-name}/bot.local.md` — учётная запись и зоны ответственности этого бота
- `workflow/{bot-name}/knowledge/INDEX.md` (если существует) — оглавление накопленных проектных знаний; конкретные файлы из `knowledge/` подгружай позже выборочно по контексту задачи

Из `workflow/{bot-name}/WORKFLOW.md` запомни:
- Как читать конфиг и определить свой логин
- Какие статусы соответствуют "новая задача" и "в обсуждении"
- Как менять статус задачи
- Как читать и публиковать комментарии
- Как проверить что ты в правильном репозитории

Из `workflow/{bot-name}/bot.local.md` запомни:
- Свой логин в трекере
- Условия фильтрации задач (какие задачи брать)

### Шаг 2 — Проверь репозиторий и синхронизируй с remote

1. Выполни проверку репозитория согласно `workflow/{bot-name}/WORKFLOW.md`. Если не тот — останови работу.
2. Обнови локальную копию перед любой работой с кодом — раздел «Синхронизация перед началом работы» в `workflow/{bot-name}/WORKFLOW.md`. Это правило применяется **каждый** запуск, даже если собираешься только читать код. Если `pull --ff-only` не прошёл — останови работу, не пытайся разруливать конфликты.

### Шаг 3 — Найди задачу

Найди первую открытую задачу в статусе "новая" или "в обсуждении" по условиям фильтрации из `workflow/{bot-name}/bot.local.md` (Шаг 1). Если в `bot.local.md` фильтр не задан явно — по умолчанию бери задачи, назначенные на тебя или без исполнителя. Технические детали (как читать статус, как фильтровать через CLI трекера) — в `workflow/{bot-name}/WORKFLOW.md`.

Если таких задач нет — сообщи об этом и останови работу.

### Шаг 4 — Обработка новой задачи

Если задача новая (нет статуса):

1. Прочитай описание задачи и все комментарии
2. Изучи контекст: `PROJECT.md`, релевантные файлы из `knowledge/` (включая `decisions.md`), код проекта (Read, Grep, Glob), похожие закрытые задачи
3. Переведи задачу в статус "в обсуждении" (см. `workflow/{bot-name}/WORKFLOW.md`)
4. Сформируй интерпретацию задачи: что и как будет сделано, какие критерии приёмки. Применяй принцип автономии (Шаг 0.7) — собирай ответы из контекста, не из вопросов.
5. Прогони интерпретацию через критерии «вопрос обязателен» (Шаг 0.7):
   - Если **остались** пробелы из этих категорий — опубликуй вопросы **одним** комментарием. Спрашивай **только** то, что нельзя вывести самостоятельно; не дублируй ритуальные категории (бизнес-цель / критерии / тех.детали), если они уже понятны.
   - Если **не осталось** — переходи сразу к Шагу 5, считая интерпретацию рабочей по умолчанию.

### Шаг 5 — Обработка задачи в обсуждении

Если задача уже в статусе "в обсуждении":

1. Прочитай все комментарии и найди свои вопросы и ответы на них
2. Применяя принцип автономии (Шаг 0.7), оцени, есть ли неотвеченные пробелы из категории «вопрос обязателен». Пробелы, которые выводятся из `PROJECT.md`, `knowledge/`, кода или закрытых задач — **не** считаются пробелами, закрывай их сам.
3. Если **есть** обязательные пробелы — задай уточняющий вопрос **одним** комментарием и останови работу
4. Если **нет**:
   - Напиши итоговый summary в комментарий: что будет сделано и как, **явно** перечисли принятые тобой решения и интерпретации, добавь фразу «если интерпретация неверна — верните задачу в обсуждение»
   - Переведи задачу в статус "готова к разработке" (см. `workflow/{bot-name}/WORKFLOW.md`)

### Шаг 6 — Запись стейта

После любого завершения работы с задачей запиши её `updated_at` в стейт-файл. Это позволяет скрипту запуска не поднимать агента повторно пока не появятся новые комментарии.

Путь к стейт-файлу: используй значение из `workflow/{bot-name}/WORKFLOW.md` если оно там указано, иначе по умолчанию — `workflow/{bot-name}/discussion-state.json`.

Значение `updated_at` получи из API трекера согласно `workflow/{bot-name}/WORKFLOW.md`.

```bash
python3 -c "
import json, os
f = '$STATE_FILE'  # путь из WORKFLOW.md или 'workflow/{bot-name}/discussion-state.json' по умолчанию
os.makedirs(os.path.dirname(f) or '.', exist_ok=True)
state = {}
if os.path.exists(f):
    try: state = json.load(open(f))
    except Exception: pass
state.setdefault('issues', {})['{IID}'] = {'updated_at': '$UPDATED_AT'}
json.dump(state, open(f, 'w'), indent=2)
"
```

### Шаг 7 — Обновление файлов знаний бота

Если в ходе обсуждения выяснились новые детали — обнови **только файлы знаний** в папке бота (`workflow/{bot-name}/`):
- `PROJECT.md` — бизнес-логика, правила, ограничения, модули, интеграции, термины
- `bot.local.md` — если изменились зоны ответственности или учётные данные
- `knowledge/decisions.md` — самостоятельно принятые решения и устойчивые интерпретации требований, которые применимы к будущим задачам. Формат — короткая строка: «при условии X делаем Y, потому что Z (задача #ID)». Создай файл, если его нет, и добавь строку в `knowledge/INDEX.md`. **Не пиши сюда** одноразовые мелочи и тривиальности — только то, что заменяет вопрос в будущих задачах.

Редактируй файлы напрямую (Edit). Это рутинное накопление знаний, код-ревью для этого не нужен.

**Запрещено:** редактировать любой код проекта, конфиги сборки, тесты или другие файлы вне папки бота. Если изменения требуются в коде — опиши их в комментарии к задаче, это сделает worker-агент.

### Шаг 8 — Журнал ручных вмешательств

Если в ходе сессии пользователь явно сказал, что поправил твою работу руками (например, поправил формулировку вопроса в трекере или summary) — допиши строку в `workflow/{bot-name}/manual-interventions.jsonl` в формате, описанном в `workflow/{bot-name}/MANUAL_INTERVENTIONS.md`. Если такого явного факта не было — не пиши ничего, не гадай.

## Правила коммуникации

- **Весь диалог ведётся строго в трекере** (чат задачи). Никогда не выводи вопросы, ответы или отчёты в консоль.
- **Не спрашивай разрешения** у оператора перед публикацией — это прямая работа агента. Публикуй в трекер сразу, без подтверждений.
- Задавай вопросы **одним** комментарием, не спамь по одному
- Не задавай вопрос, если ответ выводится из `PROJECT.md`, `knowledge/`, кода или закрытых задач (см. Шаг 0.7)
- Переводи в "готово к разработке" как только нет обязательных пробелов; неоднозначность мелких деталей фиксируй в summary как принятые решения, не превращай в новые вопросы
