Ты скилл генерации скрипта проверки и запуска агентов. Твоя задача — прочитать конфигурацию бота, понять какие агенты есть и в каких колонках они ищут задачи, и сгенерировать `workflow/{bot-name}/scripts/check.sh`. Скрипт опирается на CLI-контракт `workflow/{bot-name}/scripts/tracker` (унифицированный интерфейс к трекеру) и лежит рядом с ним в `scripts/`.

## Поведение готового скрипта

**Без аргументов** — сначала печатает шапку-помощь (краткое напоминание о доступных командах), затем таблицу состояния по всем агентам:
```
Использование:
  ./check.sh                — показать текущие статусы задач
  ./check.sh --run <agent>  — запустить конкретного агента
  ./check.sh --run all      — запустить всех агентов со статусом NEEDED=yes
  ./check.sh --help         — показать это сообщение

AGENT       TASKS  NEEDED
discussion  3      yes
worker      0      no
reviewer    1      yes
```

**`--help`** — печатает только шапку-помощь и выходит. Не обращается к трекеру.

**`--run <agent>`** — запускает конкретного агента через `claude --agent`.

**`--run all`** — запускает параллельно всех агентов у которых `NEEDED=yes`. Каждый агент работает в отдельном процессе. Поскольку разные агенты обрабатывают задачи в своих колонках, конфликтов на одну задачу не возникает.

После `--run` скрипт ждёт завершения всех запущенных и печатает итог: кто завершился успешно, кто с ошибкой.

## Шаг 2 — Изучи конфигурацию бота

Прочитай:

1. **`workflow/{bot-name}/bot.local.md`** — основной источник. Там в секциях `### Агент <имя>` описано:
   - какую колонку агент обрабатывает,
   - какой фильтр по исполнителю (на бота / без исполнителя / комбинация).
2. **`workflow/{bot-name}/WORKFLOW.md`** — для перекрёстной проверки (жизненный цикл задачи, список колонок, стейт-файлы).

**Параллельно собирай список зависимостей.** Каждый файл, который ты реально читаешь и используешь как источник содержимого `check.sh`, фиксируй в массив `DEPS` — пути относительно корня репозитория. По умолчанию туда попадают:

- `workflow/{bot-name}/bot.local.md`
- `workflow/{bot-name}/WORKFLOW.md`
- `workflow/{bot-name}/scripts/tracker` (если меняется CLI-контракт — генерируемый скрипт может звать команды/флаги, которых уже нет)

Если для конкретного бота ты прочитал ещё какие-то файлы (например, дополнительный `knowledge/<тема>.md`, чтобы понять формат стейт-файла стейтового агента) — добавь их сюда. Этот список нужен для манифеста зависимостей в Шаге 4 — `check.sh` будет проверять `mtime` каждого файла при запуске.

Для каждого агента зафиксируй:
- **Имя** — как в секции `### Агент <имя>` и как в `/agents` (для запуска через `claude --agent <имя>`).
- **Колонка** — человекочитаемое название (как в `list-columns`).
- **Фильтр** — один из вариантов:
  - `--assignee me` — только задачи на боте
  - `--assignee none` — только без исполнителя
  - `both` — и свои, и без исполнителя (две команды и объединение)
  - `all` — все задачи в колонке
- **Стейт-файл** — если агент использует стейт для дедупликации (например discussion хранит `updated_at` последнего прочитанного комментария), зафиксируй путь и структуру.

Если `bot.local.md` отсутствует или пуст — останови работу: «Запусти `configure` чтобы заполнить зоны ответственности агентов».

## Шаг 4 — Сгенерируй check.sh

Скрипт состоит из блоков в порядке ниже. Получение задач — **только через `./scripts/tracker`**, никаких прямых curl/API — это базовая гарантия унификации.

Путь генерации: `workflow/{bot-name}/scripts/check.sh` (рядом с `scripts/tracker` — оба в одной папке `scripts/`).

### Шапка

Скрипт живёт на три уровня ниже корня репозитория (`workflow/{bot-name}/scripts/check.sh` → корень), поэтому `cd` поднимается на три уровня.

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/../../.."
BOT_DIR="workflow/{bot-name}"
TRACKER="$BOT_DIR/scripts/tracker"
```

### Функция `print_help` и ранний выход по `--help`

Сразу после шапки (до любых проверок `$TRACKER`) объяви функцию `print_help` и обработай `--help`. Цель — чтобы `./check.sh --help` отвечал мгновенно и работал даже если трекер ещё не настроен (нет `scripts/tracker`). Проверка исполняемости `$TRACKER` должна идти **после** блока `--help`.

```bash
SELF="$(basename "$0")"

print_help() {
  cat <<EOF
Использование:
  ./$SELF                — показать текущие статусы задач
  ./$SELF --run <agent>  — запустить конкретного агента
  ./$SELF --run all      — запустить всех агентов со статусом NEEDED=yes
  ./$SELF --help         — показать это сообщение

Доступные агенты: <агент1>, <агент2>, ...
EOF
}

# Ранний выход по --help — без обращения к трекеру и без требования настроенного $TRACKER
if [[ "${1:-}" == "--help" || "${1:-}" == "-h" ]]; then
  print_help
  exit 0
fi

if [[ ! -x "$TRACKER" ]]; then
  echo "ERROR: $TRACKER not found or not executable. Run the carono-wf:tracker-setup skill first." >&2
  exit 2
fi
```

В строке «Доступные агенты:» перечисли имена агентов, которые дальше по скрипту поддерживаются в `--run` (те же, что в блоке «Получение задач»). Это полезная подсказка и она не зависит от рантайма — состав агентов фиксирован при генерации скрипта.

### Манифест зависимостей и проверка устаревания

Сразу после блока с проверкой `$TRACKER` (то есть до получения задач) встрой блок проверки актуальности скрипта. Логика:

- **Манифест.** При генерации (на этом самом шаге) ты для каждого файла из массива `DEPS` (см. Шаг 2) узнал текущий `mtime` (через `stat -c %Y <file>` на Linux или `stat -f %m <file>` на macOS — используй переносимый вариант `python3 -c "import os,sys; print(int(os.path.getmtime(sys.argv[1])))" <file>` или прямо `date -r <file> +%s` если есть GNU coreutils). Эти пары `путь mtime` встрой прямо в скрипт как массив строк. Не используй отдельный файл — манифест должен лежать внутри `check.sh`, чтобы при ручной правке скрипта пользователь сам видел, что именно проверяется.
- **Проверка.** При запуске скрипт проходит по манифесту, сравнивает текущий `mtime` каждого файла с записанным. Если файл новее — добавляет его в список `STALE`. Если файл удалён — добавляет в `MISSING`. После прохода: если хоть один файл в `STALE`/`MISSING` — печатает в **stderr** жёлтое (ANSI `\033[33m`) предупреждение и идёт дальше. **Без exit, без ошибки** — это just warning.
- **Где НЕ срабатывает.** Только после раннего выхода по `--help` (он всегда тихий) и до проверки `$TRACKER`. Если `$TRACKER` сам отсутствует — ошибка о нём важнее предупреждения о mtime, поэтому порядок такой: `--help` → манифест → проверка `$TRACKER` → дальше.

```bash
# --- staleness check ---
# Манифест зависимостей: пары "путь<TAB>mtime_at_generation".
# Скрипт сравнивает mtime каждого файла с записанным; если файл новее или
# удалён — печатает предупреждение, но продолжает работу.
DEPS_MANIFEST=(
  "workflow/<bot-name>/bot.local.md	<MTIME>"
  "workflow/<bot-name>/WORKFLOW.md	<MTIME>"
  "workflow/<bot-name>/scripts/tracker	<MTIME>"
  # ... остальные источники, которые скилл реально читал при генерации
)

STALE=()
MISSING=()
for entry in "${DEPS_MANIFEST[@]}"; do
  path="${entry%%	*}"
  recorded="${entry##*	}"
  if [[ ! -e "$path" ]]; then
    MISSING+=("$path")
    continue
  fi
  current=$(date -r "$path" +%s 2>/dev/null || python3 -c "import os,sys; print(int(os.path.getmtime(sys.argv[1])))" "$path")
  if [[ "$current" -gt "$recorded" ]]; then
    STALE+=("$path")
  fi
done

if [[ ${#STALE[@]} -gt 0 || ${#MISSING[@]} -gt 0 ]]; then
  YELLOW='\033[33m'
  RESET='\033[0m'
  {
    printf "%b" "${YELLOW}WARNING:${RESET} check.sh может быть устаревшим.\n"
    if [[ ${#STALE[@]} -gt 0 ]]; then
      printf "%b" "  Изменены после генерации:\n"
      for f in "${STALE[@]}"; do printf "    - %s\n" "$f"; done
    fi
    if [[ ${#MISSING[@]} -gt 0 ]]; then
      printf "%b" "  Удалены после генерации:\n"
      for f in "${MISSING[@]}"; do printf "    - %s\n" "$f"; done
    fi
    printf "%b" "  Перегенерируй: ${YELLOW}/carono-wf:checker-setup${RESET}\n"
  } >&2
fi
# --- end staleness check ---
```

**Важно:**
- В реальном скрипте `<bot-name>` подставлен на имя бота, `<MTIME>` — на целое число (epoch seconds, результат `date -r <file> +%s` на момент генерации). Никаких плейсхолдеров в готовом скрипте остаться не должно.
- Разделитель пути и mtime — символ табуляции (`\t`), потому что пробел может встретиться в путях (теоретически), а `parameter expansion` `${entry%%TAB*}` / `${entry##*TAB}` корректно режет по табу. Если шелл затрудняет работу с табом — допустимо использовать `|` или другой явно нестандартный символ, но не пробел.
- Если массив `DEPS` пуст (необычный случай) — блок проверки не генерируй вовсе.

### Получение задач для каждого агента

Для агента с фильтром **`--assignee me`**:
```bash
TASKS_<АГЕНТ>=$("$TRACKER" list-tasks --column "<колонка>" --assignee me)
```

Для агента с фильтром **`--assignee none`**:
```bash
TASKS_<АГЕНТ>=$("$TRACKER" list-tasks --column "<колонка>" --assignee none)
```

Для агента с фильтром **`both`** (на боте ИЛИ без исполнителя) — два запроса + объединение по `id`:
```bash
TASKS_<АГЕНТ>=$(
  {
    "$TRACKER" list-tasks --column "<колонка>" --assignee me
    "$TRACKER" list-tasks --column "<колонка>" --assignee none
  } | jq -s 'add | unique_by(.id)'
)
```

Для агента с фильтром **`all`** (все задачи колонки):
```bash
TASKS_<АГЕНТ>=$("$TRACKER" list-tasks --column "<колонка>")
```

### Подсчёт задач

Для агента **без стейт-файла**:
```bash
COUNT_<АГЕНТ>=$(echo "$TASKS_<АГЕНТ>" | jq 'length')
NEEDED_<АГЕНТ>="no"
[[ "$COUNT_<АГЕНТ>" -gt 0 ]] && NEEDED_<АГЕНТ>="yes"
```

Для агента **со стейт-файлом** (пример: discussion хранит `updated_at` последнего сообщения в чате задачи):
```bash
STATE_<АГЕНТ>="$BOT_DIR/<имя-стейта>.json"

COUNT_<АГЕНТ>_TOTAL=$(echo "$TASKS_<АГЕНТ>" | jq 'length')

COUNT_<АГЕНТ>_NEW=$(python3 - <<PY
import json, os, subprocess
tasks = json.loads('''$TASKS_<АГЕНТ>''')
state = {}
p = "$STATE_<АГЕНТ>"
if os.path.exists(p):
    try: state = json.load(open(p))
    except Exception: pass
issues = state.get("issues", {})
new = 0
for t in tasks:
    tid = str(t["id"])
    comments = subprocess.check_output(["$TRACKER", "list-comments", t["id"]])
    last = (json.loads(comments) or [{}])[-1].get("id", "")
    saved = issues.get(tid, {}).get("updated_at", "")
    # normalize — state may hold str or int, API returns int
    if str(saved) != str(last):
        new += 1
print(new)
PY
)

NEEDED_<АГЕНТ>="no"
[[ "$COUNT_<АГЕНТ>_NEW" -gt 0 ]] && NEEDED_<АГЕНТ>="yes"
```

**Примечание.** В стейтовых агентах структура `state.json` диктуется самим агентом (смотри `workflow/{bot-name}/WORKFLOW.md`, раздел про стейт-файл). В примере выше — формат для discussion, где ключ в `issues` — ID задачи, значение `updated_at` — ID последнего сообщения. Если в твоём проекте формат другой — адаптируй.

### Режим status (по умолчанию)

```bash
MODE="${1:-status}"

if [[ "$MODE" == "status" ]]; then
  print_help
  echo
  printf "%-15s %-12s %s\n" "AGENT" "TASKS" "NEEDED"
  printf "%-15s %-12s %s\n" "<агент1>" "$COUNT_<АГЕНТ1>" "$NEEDED_<АГЕНТ1>"
  # ... для каждого агента
  exit 0
fi
```

`print_help` вызывается перед таблицей — чтобы при каждом запуске `./check.sh` без аргументов разработчик сразу видел доступные команды.

Для стейтовых агентов в колонке `TASKS` выводи `NEW (из TOTAL)`, например `2 (из 5)`:
```bash
printf "%-15s %-12s %s\n" "discussion" "$COUNT_discussion_NEW (из $COUNT_discussion_TOTAL)" "$NEEDED_discussion"
```

### Режим `--run`

```bash
if [[ "$MODE" == "--run" ]]; then
  TARGET="${2:-}"
  if [[ -z "$TARGET" ]]; then
    echo "Использование: $0 --run <agent|all>" >&2
    exit 1
  fi

  mkdir -p "$BOT_DIR/logs"

  run_agent() {
    local agent="$1"
    echo "[start] $agent"
    if claude -p "Выполни свои задачи" --agent "$agent" \
         > "$BOT_DIR/logs/${agent}-$(date +%s).log" 2>&1; then
      echo "[ok]    $agent"
    else
      echo "[fail]  $agent (см. $BOT_DIR/logs/)"
    fi
  }

  if [[ "$TARGET" == "all" ]]; then
    PIDS=()
    for pair in \
      "<агент1>:$NEEDED_<АГЕНТ1>" \
      "<агент2>:$NEEDED_<АГЕНТ2>"; do
      agent="${pair%%:*}"
      needed="${pair##*:}"
      if [[ "$needed" == "yes" ]]; then
        run_agent "$agent" &
        PIDS+=($!)
      fi
    done
    for pid in "${PIDS[@]}"; do wait "$pid"; done
  else
    run_agent "$TARGET"
  fi

  exit 0
fi

print_help >&2
exit 1
```

**Имена агентов в `claude --agent`.** Если плагин уже поставлен, имя — с префиксом плагина (например `carono-wf:discussion`). Если агенты лежат локально в `.claude/agents/` — без префикса (просто `discussion`). Определи способ установки по наличию `.claude-plugin/` или `.claude/agents/` в проекте и используй соответствующий формат.

## Шаг 6 — Dry-run

1. Запусти `./workflow/{bot-name}/scripts/check.sh --help` и убедись что шапка-помощь выводится и скрипт завершается с кодом 0 без обращения к трекеру и **без предупреждения об устаревании** (для `--help` блок staleness не запускается).
2. Запусти `./workflow/{bot-name}/scripts/check.sh` без аргументов и покажи пользователю реальный вывод — шапку-помощь + таблицу состояния. Если видишь ошибку (например `list-tasks: column not found`) — значит в генерации перепутались названия колонок, чини и повторяй до чистого запуска.
3. Сразу после генерации предупреждение об устаревании **не должно** срабатывать — `mtime` всех зависимостей записан только что. Если предупреждение появилось — проверь, что записал в манифест именно `mtime` на момент генерации, а не что-то ранее.

## Шаг 7 — Итог

Сообщи пользователю:
- Какие агенты в скрипте и с какими колонками/фильтрами
- Показанный вывод `./check.sh` (шапку-помощь + таблицу)

## Правила

- **Только CLI.** Никаких прямых HTTP-запросов в `check.sh`, только `./scripts/tracker <cmd>`. Это инвариант.
- **Никаких креденшиалов в скрипте.** Токен живёт только в `.env` трекера, `check.sh` его не читает и не передаёт.
- **Перегенерация, а не правка.** Если обнаружены расхождения — пиши целиком через Write, частичный Edit на bash хрупок.
- **Манифест зависимостей встроен в скрипт.** Список `путь mtime` хранится прямо в `check.sh`, без отдельного стейт-файла. При запуске скрипт сравнивает с текущим `mtime`, при устаревании — жёлтое предупреждение в stderr и продолжение работы (не падение, не ненулевой exit).
- **Не создавай `logs/` заранее** — скрипт создаёт её сам при `--run`.
- **Stateful-агенты.** Если агент использует стейт — уточни у пользователя или прочитай в `WORKFLOW.md` точный формат, не угадывай. В случае сомнений сделай без стейта (`COUNT` = всем задачам в колонке) и поставь `*(TODO: интегрировать стейт)*` комментарием.
