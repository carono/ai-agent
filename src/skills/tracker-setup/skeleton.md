# Скелет скрипта `tracker` и правила генерации

Этот документ — **правила генерации**, а не готовый шаблон. Скилл `tracker-setup` каждый раз пишет скрипт **с нуля**, делая живые пробы API трекера и составляя маппинги по фактическим ответам, а не по памяти. Готовых копипастов нет — API меняются, а шаблоны устаревают незаметно.

Итоговый скрипт обязан 1-в-1 соответствовать `cli-contract.md` — контракту входа/выхода. Контракт не меняется, меняется только то, **как** скрипт его реализует под конкретный API.

---

## 1. Выбор языка

Решается **до** начала генерации.

- **Bash + curl + jq** — по умолчанию. Подходит для:
  - REST API с Bearer-токеном или Basic-auth
  - JSON-тела запросов
  - Простая пагинация (offset/limit, cursor)
- **Python 3 + requests** — если трекер требует:
  - OAuth-потока вместо простого токена
  - накопительной/нетривиальной пагинации (GraphQL, курсоры, rate-limit-паузы)
  - работы с MCP-сервером
  - бинарных вложений
  - нетривиальных трансформаций (деревья подзадач, агрегация нескольких источников)
- **Node.js** — только если нужен официальный SDK/MCP трекера, и других вариантов нет.

Скажи пользователю явно: «Генерирую `tracker` на `<bash|python>`, потому что `<одна фраза>`».

---

## 2. Обязательная структура скрипта

Любой сгенерированный скрипт должен содержать эти блоки **в этом порядке**. Менять нельзя — это контракт по устройству самого файла.

### 2.1 Shebang + strict mode

Bash: `#!/usr/bin/env bash` + `set -euo pipefail` в начале.
Python: `#!/usr/bin/env python3` + `import sys` и явные `sys.exit(code)`.

### 2.2 Определение собственных путей

Пути вычисляются **от `$0` / `__file__`**, а не от CWD. Агент может запускать скрипт из любой директории — CWD не гарантируется.

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
BOT_DIR="$(cd "$SCRIPT_DIR/.." && pwd)"
ENV_FILE="$BOT_DIR/.env"
```

### 2.3 Загрузка `.env`

Только `workflow/{bot-name}/.env`. Никаких глобальных источников (`~/.bashrc`, хост-переменные окружения). Отсутствие файла → `exit 2` с понятным сообщением в stderr.

### 2.4 Проверка обязательных переменных

Сразу после загрузки `.env` — проверка, что все требуемые ключи присутствуют. Имена ключей берутся из `cli-contract.md` (`TRACKER_API_KEY`, `TRACKER_BASE_URL`, …) плюс специфичные для трекера (`YOUGILE_TEAM_ID`, `JIRA_EMAIL`, `GITHUB_OWNER` и т.п.). Отсутствующая переменная → `exit 2`.

### 2.5 Проверка зависимостей

Bash-вариант: `curl`, `jq`. Python-вариант: `requests` (если без него никак) — проверка через `import`.

### 2.6 HTTP-хелперы

Одна-две функции (`api_get`, `api_post`, `api_put`, `api_delete`), которые:
- добавляют заголовки авторизации (Bearer / Basic / кастомный)
- сериализуют JSON-тело
- возвращают ненулевой код при non-2xx ответе (с текстом ошибки в stderr)

### 2.7 Резолверы имён → ID

Агент передаёт **названия** (колонок, пользователей) — скрипт превращает в ID трекера. Минимум: `resolve_column_id(name)`, `resolve_user_id(login)`. Реализация через API-запросы (`list-columns`, `list-users`), можно с лёгким кешем в рантайме одного вызова.

### 2.8 Dispatcher

Bash: `case` по первому позиционному аргументу. Python: словарь `{"команда": функция}`.

### 2.9 `usage()`

Выводит в stderr список команд и опций. Вызывается по `--help`, `-h`, при неизвестной команде, при отсутствии обязательных аргументов. Ссылается на `cli-contract.md`.

## 3. Процесс генерации — **главное правило**

Для каждой команды из `cli-contract.md` (их 11: `whoami`, `list-columns`, `list-tasks`, `get-task`, `list-comments`, `comment`, `move-task`, `assign-task`, `create-task`, `delete-task`, `list-users`):

### Шаг A. Найди endpoint в API трекера

По документации или по логике: прочитал → понял, какой HTTP-метод и путь нужны. Не помнишь — прочти документацию API, не угадывай.

### Шаг B. Сделай пробный запрос через `curl`

```bash
curl -sS -H "Authorization: Bearer $TOKEN" "$BASE_URL/<endpoint>" | jq .
```

Для операций, создающих/меняющих данные — пробуй на тестовой задаче или передавай параметры в dry-run режиме, если API это поддерживает. Для `create-task` — создай тестовую, запомни её ID, удали в конце генерации.

### Шаг C. Посмотри живую структуру ответа

Фиксируй реальные имена полей. **Не полагайся на память.** Частые ловушки:
- `created_at` vs `created` vs `createdAt` vs `timestamp`
- id как UUID vs integer vs prefixed string (`WF-21`)
- assignee как `[id]` vs `{id: ...}` vs `{users: [...]}`
- pagination: `{content, next}` vs `{items, cursor}` vs `{results, page_info}`

### Шаг D. Напиши маппинг ответа API → формат контракта

Используй `jq` (bash) или генератор выражений (python). Имена полей в маппинге — **ровно те, что увидел в Шаге C**, не «вероятные».

### Шаг E. Проверь что вывод команды соответствует `cli-contract.md`

- JSON валиден: `./tracker <cmd> | jq type`
- Структура совпадает: если контракт требует массив объектов с полями `{id, number, title, column, ...}` — значит они все должны присутствовать, даже если пустые строки/null.
- Коды возврата: `0` на успех, `3` на not-found, `4` на HTTP-ошибку.

### Шаг F. Идентичный подход для следующей команды

Повтори Шаги A–E для каждой из 11 команд. Не копируй маппинг из одной команды в другую — даже в одном API поля могут отличаться между эндпоинтами (например `timestamp` в задачах, но `created` в комментариях).

## 4. Что НЕЛЬЗЯ делать

- **Полагаться на память об API.** Всегда — проба, потом код. Это инвариант.
- **Копировать маппинг из предыдущих проектов** без проверки. YouGile мог за неделю поменять имя поля — и ни один старый скрипт об этом не узнает.
- **Писать интерактивный ввод** в скрипте. Всегда неинтерактивный: аргументы + stdin при необходимости.
- **Логировать токен/креденшиалы** в stdout, stderr, файлы. Даже при `set -x` — оборачивай debug-режим флагом.
- **Использовать глобальные env** (`$HOME/.netrc`, `~/.bashrc`) — только `workflow/{bot-name}/.env`.
- **Писать человекочитаемый текст в stdout.** Stdout — только JSON по контракту. Диагностика — в stderr.
- **Смешивать команды** — у каждой свой формат вывода.
- **Хардкодить ID колонок/пользователей** в коде. Все маппинги — рантайм через резолверы.
- **Использовать `eval`** на пользовательских строках. Комментарии — через stdin или файл (`comment ... --file`).

---

## 5. Формат вывода — напоминание

- stdout — JSON, соответствующий схеме команды из `cli-contract.md`
- stderr — человекочитаемые ошибки
- Коды возврата:
  - `0` — успех
  - `1` — общая ошибка (неверный аргумент, неизвестная команда)
  - `2` — проблема с конфигурацией / креденшиалами / зависимостями
  - `3` — ресурс не найден
  - `4` — сетевая / API-ошибка

---

## 6. Spec-только трекеры (MCP без REST)

Если у трекера нет прямого REST, а только MCP-сервер — **не генерируй скрипт**. Сообщи пользователю:

> У `<трекер>` нет прямого REST API — скрипт `scripts/tracker` не нужен. Агенты будут ходить в трекер через MCP-сервер напрямую. `WORKFLOW.md` оформлю под MCP-вариант (но этот скилл такое не делает — сейчас он заточен под CLI-контракт).

Это редкий случай; большинство трекеров имеют REST параллельно с MCP.

## 7. Smoke-test после генерации

**Недостаточно** прогнать `whoami` + `list-columns` + `list-users`. Проверь **все 11 команд** — хотя бы в read-режиме:

| Команда | Smoke-проверка |
|---|---|
| `whoami` | валидный JSON с непустым `login` |
| `list-columns` | массив объектов `{id, name, position}`, длина > 0 |
| `list-tasks --column <первая колонка>` | массив объектов, структура соответствует контракту |
| `get-task <id>` | взять id из `list-tasks`, вернуть полный объект |
| `list-comments <id>` | массив (возможно пустой) объектов `{id, author, text, created_at}` |
| `comment <тестовая задача> "test"` | создать/удалить тестовую задачу — написать в неё комментарий, проверить через `list-comments` что он появился |
| `move-task <тест> --column <другая>` | на тестовой задаче, вернуть обратно |
| `assign-task <тест> --user me` / `none` | на тестовой |
| `create-task --title "smoke" --column <первая>` | создать, проверить через `get-task` |
| `list-users` | массив > 0 |
| `delete-task <тест>` | убрать тестовую задачу |

Порядок: создать тестовую задачу → прогнать все мутирующие операции на ней → удалить. Это и smoke-test скрипта, и проверка что в реальном API поля совпадают с твоими маппингами.

Если хоть одна проверка провалилась — покажи stderr пользователю, ставь тикет «поправить эту команду», доработай, повтори тест. **Не завершай работу скилла, пока все 11 не зелёные.**
