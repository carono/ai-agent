# Скелет скрипта `tracker` и правила генерации

Этот документ — **правила генерации**, а не готовый шаблон. Скилл `tracker-setup` каждый раз пишет скрипт **с нуля**, делая живые пробы API трекера и составляя маппинги по фактическим ответам, а не по памяти. Готовых копипастов нет — API меняются, а шаблоны устаревают незаметно.

Итоговый скрипт обязан 1-в-1 соответствовать `cli-contract.md` — контракту входа/выхода. Контракт не меняется, меняется только то, **как** скрипт его реализует под конкретный API.

---

## 1. Выбор языка

Решается **до** начала генерации.

- **Bash + curl + jq** — по умолчанию. Подходит для:
  - REST API с Bearer-токеном или Basic-auth
  - JSON-тела запросов
  - Простая пагинация (offset/limit, cursor)
- **Python 3 + requests** — если трекер требует:
  - OAuth-потока вместо простого токена
  - накопительной/нетривиальной пагинации (GraphQL, курсоры, rate-limit-паузы)
  - работы с MCP-сервером
  - бинарных вложений
  - нетривиальных трансформаций (деревья подзадач, агрегация нескольких источников)
- **Node.js** — только если нужен официальный SDK/MCP трекера, и других вариантов нет.

Скажи пользователю явно: «Генерирую `tracker` на `<bash|python>`, потому что `<одна фраза>`».

---

## 2. Обязательная структура скрипта

Любой сгенерированный скрипт должен содержать эти блоки **в этом порядке**. Менять нельзя — это контракт по устройству самого файла.

### 2.1 Shebang + strict mode

Bash: `#!/usr/bin/env bash` + `set -euo pipefail` в начале.
Python: `#!/usr/bin/env python3` + `import sys` и явные `sys.exit(code)`.

### 2.2 Определение собственных путей

Пути вычисляются **от `$0` / `__file__`**, а не от CWD. Агент может запускать скрипт из любой директории — CWD не гарантируется.

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
BOT_DIR="$(cd "$SCRIPT_DIR/.." && pwd)"
ENV_FILE="$BOT_DIR/.env"
```

### 2.3 Загрузка `.env`

Только `workflow/{bot-name}/.env`. Никаких глобальных источников (`~/.bashrc`, хост-переменные окружения). Отсутствие файла → `exit 2` с понятным сообщением в stderr.

### 2.4 Проверка обязательных переменных

Сразу после загрузки `.env` — проверка, что все требуемые ключи присутствуют. Имена ключей берутся из `cli-contract.md` (`TRACKER_API_KEY`, `TRACKER_BASE_URL`, …) плюс специфичные для трекера (`YOUGILE_TEAM_ID`, `JIRA_EMAIL`, `GITHUB_OWNER` и т.п.). Отсутствующая переменная → `exit 2`.

### 2.5 Проверка зависимостей

Bash-вариант: `curl`, `jq`. Python-вариант: `requests` (если без него никак) — проверка через `import`.

### 2.6 HTTP-хелперы

Одна-две функции (`api_get`, `api_post`, `api_put`, `api_delete`), которые:
- добавляют заголовки авторизации (Bearer / Basic / кастомный)
- сериализуют JSON-тело
- возвращают ненулевой код при non-2xx ответе (с текстом ошибки в stderr)

### 2.7 Резолверы имён → ID

Агент передаёт **названия** (колонок, пользователей) — скрипт превращает в ID трекера. Минимум: `resolve_column_id(name)`, `resolve_user_id(login)`. Реализация через API-запросы (`list-columns`, `list-users`), можно с лёгким кешем в рантайме одного вызова.

### 2.8 Dispatcher

Bash: `case` по первому позиционному аргументу. Python: словарь `{"команда": функция}`.

### 2.9 `usage()`

Выводит в stderr список команд и опций. Вызывается по `--help`, `-h`, при неизвестной команде, при отсутствии обязательных аргументов. Ссылается на `cli-contract.md`.

## 3. Pre-flight чек-лист — **до** генерации скрипта

Этот чек-лист проходится **один раз**, до записи `.env` и до per-command discovery (раздел 4). Цель — убедиться, что фундаментальные возможности трекера, на которые опирается CLI-контракт, реально доступны в этом инстансе. Каждый пункт — пара «проба → ожидаемый ответ → что зафиксировать в маппинге CLI». Если проба провалилась — не маскируй заглушкой и не молчи: явно подними флаг пользователю с описанием, что именно не работает, и предложи решение (другой токен, другие права, ручной обход, отказ от настройки).

Все ID и поля, найденные на этом шаге, идут в `.env` (если это креденшиалы/координаты) или в комментарии маппинга при per-command discovery (если это имена полей/ключей).

### 3.1 Аутентификация

- **Проба.** Минимальный авторизованный запрос (например `GET /me`, `GET /user`, `GET /projects/<id>`) с переданным токеном.
- **Ожидание.** HTTP 200 + JSON с пользователем/проектом. 401/403 → токен недействителен или прав не хватает.
- **В маппинг.** Заголовок авторизации (`Authorization: Bearer ...` / `Basic ...` / кастомный) и его точный формат.
- **Флаг при провале.** Сообщи пользователю: «Токен невалиден / прав не хватает на чтение пользователя. Нужен токен с правами на чтение задач, комментариев, участников. Где брать — в настройках профиля / приложений `<трекер>`.» Не продолжай.

### 3.2 Структура доски

- **Проба.** Запрос списка колонок/статусов (`GET /columns?boardId=...`, `GET /repos/.../labels`, `GET /project/<id>/statuses`).
- **Ожидание.** Непустой массив с идентификаторами + именами + позицией в lifecycle.
- **В маппинг.** Имена полей (`id`, `name`, `position` / `order` / `index`), способ привязки к доске/проекту, нужно ли отдельно фильтровать «архивные» колонки.
- **Флаг при провале.** Если у трекера нет колонок (Jira со статусами / GitHub Issues с лейблами) — подтверди что ты будешь маппить статусы/лейблы в формат `list-columns`. Если нет ни того ни другого — пиши пользователю: «У `<трекер>` не нашлось ни колонок, ни статусов, ни лейблов в проекте `<id>`. CLI-контракт `list-columns` для этого трекера реализуем не будет, агенты не смогут различать стадии задач. Останавливаю настройку.»

### 3.3 Идентификация задач

- **Проба.** Прочитай одну реальную задачу (любую, существующую) — посмотри какие ID она несёт.
- **Ожидание.** Понятно, какое поле — внутренний `id` (UUID/integer), какое — человекочитаемый `number` (`WF-21`, `#42`). Если они совпадают — это тоже ответ.
- **В маппинг.** Пара полей под `id` и `number` контракта; правила сборки `url` (UI-ссылка) из найденных полей.
- **Стабильность номера.** Удостоверься (по документации или дополнительной пробой), что `number` не меняется при перемещении задачи между колонками/проектами. Если меняется — зафиксируй это ограничение в комментарии скрипта, чтобы агенты знали, что ссылки могут протухать.
- **Флаг при провале.** «Не нашёл человекочитаемого номера задачи. Буду использовать внутренний `id` в обоих полях `id`/`number`. Ссылки на задачи в комментариях/коммитах будут содержать UUID — это снижает читаемость, но контракт сохраняется.» Продолжай только если пользователь подтвердил.

### 3.4 Назначения (assignees)

- **Проба.** Прочитай задачу с известным исполнителем + задачу без исполнителя. Сравни структуру поля.
- **Ожидание.** Понятно, как обозначается «нет исполнителя» (пустой массив / `null` / отсутствие ключа), как добавить/снять исполнителя (PUT/PATCH с массивом id, отдельный endpoint), поддерживается ли несколько исполнителей.
- **В маппинг.** Имя поля (`assignees` / `assignee` / `users`), формат значения (массив ID / массив объектов), endpoint и метод смены.
- **`me` / `none`.** Зафиксируй, как резолвить эти спец-значения: `me` → ID владельца токена (полученный в 3.1), `none` → формат «снять» (пустой массив / `null`).
- **Флаг при провале.** «Назначения в `<трекер>` не управляются через API (только через UI) / поддерживается только один исполнитель. Команда `assign-task` будет работать ограниченно: `<точное описание ограничения>`.» Продолжай только если пользователь готов на эти ограничения.

### 3.5 Комментарии

- **Проба.** Прочитай комментарии существующей задачи + создай тестовый комментарий с многострочным markdown-текстом, содержащим пустые строки и спецсимволы (` `* _ ` ). Удали тестовый комментарий или оставь как есть, если API не умеет удалять.
- **Ожидание.** Список читается, новый комментарий создаётся в нужной задаче, многострочный текст сохраняется без обрезания/искажения, markdown отображается. Видно поля `id`, `author`, `text`, `created_at`.
- **В маппинг.** Имена полей в ответе (`text` / `body` / `content`, `created_at` / `created` / `timestamp`, `author` / `user`), endpoint POST для создания, ограничения по длине.
- **Флаг при провале.** «`<трекер>` не сохраняет markdown / режет многострочные комментарии / не возвращает author. Агенты `discussion` и `worker` будут писать однострочный plain-text. Подтверди что готов с этим жить.»

### 3.6 Смена статуса

- **Проба.** На тестовой задаче переведи её в другую колонку/статус, прочитай обратно — увидь что значение применилось.
- **Ожидание.** Понятно, как именно меняется статус (PUT/PATCH задачи с новым `column_id` / отдельный endpoint `/transitions` / просто смена `status`).
- **В маппинг.** Endpoint, метод, имя передаваемого поля. Зафиксируй атомарность: если для перехода нужен workflow transition (как в Jira) — список разрешённых переходов и как его получить.
- **Флаг при провале.** «`<трекер>` не позволяет произвольно менять статус — только по разрешённым transition. Команда `move-task` будет проверять допустимость перехода и возвращать ошибку `1` если переход не разрешён.» Если переходы вообще недоступны — стоп.

### 3.7 Ограничения API

- **Проба.** Сделай несколько (3–5) последовательных запросов подряд — посмотри есть ли rate-limit заголовки (`X-RateLimit-*`, `Retry-After`). Запроси список задач без параметров — посмотри пагинацию (cursor / offset / page).
- **Ожидание.** Понятно, есть ли лимиты на запросы в минуту, как устроена пагинация, в каком формате возвращаются ошибки (`{"error": "..."}` / `{"errors": [...]}` / простой текст).
- **В маппинг.** Параметры пагинации (`limit`+`offset` / `page`+`per_page` / `cursor`), правила сборки следующей страницы, формат разбора ошибок в HTTP-хелпере (`api_get`/`api_post`).
- **Флаг при провале.** «У `<трекер>` жёсткий rate-limit (например 60 запросов/минуту). Smoke-test и list-* команды будут работать медленно / с задержками. Для частых проверок (`check.sh`) лучше использовать кэш.» Это предупреждение, не стоп.

### 3.7.5 Приоритет задач

- **Проба.** Прочитай несколько (3–5) задач с разной «важностью» — посмотри, есть ли у трекера понятие приоритета и как оно представлено. Возможные варианты:
  - **Отдельное поле задачи** (Jira `fields.priority.name`, Linear `priority`, Asana `custom_fields`).
  - **Лейблы/теги с префиксом** (GitHub Issues: лейблы `priority:high`, `priority:low`).
  - **Стикеры/маркеры на доске** (YouGile: на проекте задаются стикеры с ролью «приоритет», на задаче лежит ID конкретного значения стикера).
  - **Колонка как приоритет** — отдельной сущности нет, важность кодируется столбцом доски. В таком случае приоритета как поля нет.
- **Ожидание.** Понятно, можно ли стабильно вытащить «приоритет задачи» в виде короткой строки (`"high"` / `"P0"` / `"критично"`) или, если ничего такого нет, явно зафиксировать что `priority` всегда `null`.
- **В маппинг.** Способ извлечения значения для контракта `list-tasks`/`get-task`:
  - имя поля API,
  - правило преобразования в строку (если это объект — какое поле объекта берём; если это лейбл с префиксом — как срезаем префикс; если это ID стикера — где взять human-readable значение),
  - что отдавать когда у задачи приоритет не выставлен (в большинстве трекеров — `null`).
- **Флаг при провале.** Если у трекера нет ни поля, ни лейблов, ни стикеров под приоритет — это **не ошибка**: фиксируешь в маппинге `priority: null` для всех задач, в сводке 3.9 пишешь «Приоритет: не поддерживается трекером, всегда null». Останавливаться не надо. Если приоритет есть, но выражен через колонки доски — также `priority: null`, и в комментарии маппинга поясни «приоритет в этом проекте кодируется колонкой, отдельного поля нет».

### 3.8 Идентификация бота

- **Проба.** В ответе `whoami` (или эквивалента из 3.1) посмотри, в каком формате возвращается логин бота: `login`, `email`, `realName`, `username`. Совпадает ли он с тем, что пользователь видит в UI как «исполнителя»?
- **Ожидание.** Понятно, какое поле API соответствует «человеческому имени», под которым бот будет отображаться в задачах. Это значение пойдёт в `TRACKER_BOT_LOGIN`.
- **В маппинг.** Имя поля (`login` / `email` / `realName`), формат фильтра `--assignee me` (по этому же полю или по ID).
- **Флаг при провале.** «`whoami` не возвращает имя бота / возвращает только UUID. В `bot.local.md` запишу UUID — это значит, в `--assignee` агентам тоже придётся передавать UUID. Подтверди.»

### 3.9 Сводный итог pre-flight

После прохождения 3.1–3.8 — выведи пользователю краткую сводку **одним сообщением** перед записью `.env` и скрипта:

```
Pre-flight для <трекер>:
- Аутентификация: OK (токен валиден, права достаточны)
- Доска: 5 колонок найдено
- Номер задачи: WF-<n> (поле idTaskProject)
- Назначения: массив user-id (`assignees`)
- Комментарии: markdown сохраняется
- Смена статуса: PATCH /tasks/<id> {columnId}
- Приоритет: стикер «Приоритет» на проекте, значения «Низкий/Средний/Высокий» (или: «не поддерживается, всегда null»)
- Rate-limit: 60 req/min
- Логин бота: "Carono AI" (поле realName)
```

Если хоть один пункт под флагом «частично работает / не работает» — упомяни это явно. Записывать `.env` и генерировать скрипт можно только когда пользователь подтвердил сводку.

## 4. Процесс генерации — **главное правило**

Для каждой команды из `cli-contract.md` (их 11: `whoami`, `list-columns`, `list-tasks`, `get-task`, `list-comments`, `comment`, `move-task`, `assign-task`, `create-task`, `delete-task`, `list-users`):

### Шаг A. Найди endpoint в API трекера

По документации или по логике: прочитал → понял, какой HTTP-метод и путь нужны. Не помнишь — прочти документацию API, не угадывай.

### Шаг B. Сделай пробный запрос через `curl`

```bash
curl -sS -H "Authorization: Bearer $TOKEN" "$BASE_URL/<endpoint>" | jq .
```

Для операций, создающих/меняющих данные — пробуй на тестовой задаче или передавай параметры в dry-run режиме, если API это поддерживает. Для `create-task` — создай тестовую, запомни её ID, удали в конце генерации.

### Шаг C. Посмотри живую структуру ответа

Фиксируй реальные имена полей. **Не полагайся на память.** Частые ловушки:
- `created_at` vs `created` vs `createdAt` vs `timestamp`
- id как UUID vs integer vs prefixed string (`WF-21`)
- assignee как `[id]` vs `{id: ...}` vs `{users: [...]}`
- pagination: `{content, next}` vs `{items, cursor}` vs `{results, page_info}`

### Шаг D. Напиши маппинг ответа API → формат контракта

Используй `jq` (bash) или генератор выражений (python). Имена полей в маппинге — **ровно те, что увидел в Шаге C**, не «вероятные».

### Шаг E. Проверь что вывод команды соответствует `cli-contract.md`

- JSON валиден: `./tracker <cmd> | jq type`
- Структура совпадает: если контракт требует массив объектов с полями `{id, number, title, column, ...}` — значит они все должны присутствовать, даже если пустые строки/null.
- Коды возврата: `0` на успех, `3` на not-found, `4` на HTTP-ошибку.

### Шаг F. Идентичный подход для следующей команды

Повтори Шаги A–E для каждой из 11 команд. Не копируй маппинг из одной команды в другую — даже в одном API поля могут отличаться между эндпоинтами (например `timestamp` в задачах, но `created` в комментариях).

## 5. Что НЕЛЬЗЯ делать

- **Полагаться на память об API.** Всегда — проба, потом код. Это инвариант.
- **Копировать маппинг из предыдущих проектов** без проверки. YouGile мог за неделю поменять имя поля — и ни один старый скрипт об этом не узнает.
- **Писать интерактивный ввод** в скрипте. Всегда неинтерактивный: аргументы + stdin при необходимости.
- **Логировать токен/креденшиалы** в stdout, stderr, файлы. Даже при `set -x` — оборачивай debug-режим флагом.
- **Использовать глобальные env** (`$HOME/.netrc`, `~/.bashrc`) — только `workflow/{bot-name}/.env`.
- **Писать человекочитаемый текст в stdout.** Stdout — только JSON по контракту. Диагностика — в stderr.
- **Смешивать команды** — у каждой свой формат вывода.
- **Хардкодить ID колонок/пользователей** в коде. Все маппинги — рантайм через резолверы.
- **Использовать `eval`** на пользовательских строках. Комментарии — через stdin или файл (`comment ... --file`).

---

## 6. Формат вывода — напоминание

- stdout — JSON, соответствующий схеме команды из `cli-contract.md`
- stderr — человекочитаемые ошибки
- Коды возврата:
  - `0` — успех
  - `1` — общая ошибка (неверный аргумент, неизвестная команда)
  - `2` — проблема с конфигурацией / креденшиалами / зависимостями
  - `3` — ресурс не найден
  - `4` — сетевая / API-ошибка

## 7. Spec-только трекеры (MCP без REST)

Если у трекера нет прямого REST, а только MCP-сервер — **не генерируй скрипт**. Сообщи пользователю:

> У `<трекер>` нет прямого REST API — скрипт `scripts/tracker` не нужен. Агенты будут ходить в трекер через MCP-сервер напрямую. `WORKFLOW.md` оформлю под MCP-вариант (но этот скилл такое не делает — сейчас он заточен под CLI-контракт).

Это редкий случай; большинство трекеров имеют REST параллельно с MCP.

## 8. Smoke-test после генерации

**Недостаточно** прогнать `whoami` + `list-columns` + `list-users`. Проверь **все 11 команд** — хотя бы в read-режиме:

| Команда | Smoke-проверка |
|---|---|
| `whoami` | валидный JSON с непустым `login` |
| `list-columns` | массив объектов `{id, name, position}`, длина > 0 |
| `list-tasks --column <первая колонка>` | массив объектов, структура соответствует контракту; в каждом объекте есть поле `priority` (строка или `null`) |
| `get-task <id>` | взять id из `list-tasks`, вернуть полный объект; поле `priority` присутствует (строка или `null`) |
| `list-comments <id>` | массив (возможно пустой) объектов `{id, author, text, created_at}` |
| `comment <тестовая задача> "test"` | создать/удалить тестовую задачу — написать в неё комментарий, проверить через `list-comments` что он появился |
| `move-task <тест> --column <другая>` | на тестовой задаче, вернуть обратно |
| `assign-task <тест> --user me` / `none` | на тестовой |
| `create-task --title "smoke" --column <первая>` | создать, проверить через `get-task` |
| `list-users` | массив > 0 |
| `delete-task <тест>` | убрать тестовую задачу |

Порядок: создать тестовую задачу → прогнать все мутирующие операции на ней → удалить. Это и smoke-test скрипта, и проверка что в реальном API поля совпадают с твоими маппингами.

Если хоть одна проверка провалилась — покажи stderr пользователю, ставь тикет «поправить эту команду», доработай, повтори тест. **Не завершай работу скилла, пока все 11 не зелёные.**
