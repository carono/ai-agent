# Контракт CLI `tracker`

Этот документ — **обязательный контракт** на поведение скрипта `workflow/{bot-name}/scripts/tracker`. Все реализации (под YouGile, GitHub, GitLab, Jira, ...) должны соблюдать этот контракт один-в-один: только тогда агенты могут работать с любым трекером единообразно.

Если при генерации скрипта какая-то команда физически не реализуема в конкретном трекере — всё равно реализуй её с корректным ответом (см. раздел «Недоступные операции»), не выкидывай.

---

## Общие правила

### Запуск

```
./scripts/tracker <команда> [аргументы]
```

Скрипт исполняемый (`chmod +x`), с shebang в первой строке (`#!/usr/bin/env bash` или `#!/usr/bin/env python3` — по выбору генератора).

### Переменные окружения

Скрипт читает креденшиалы **строго** из `workflow/{bot-name}/.env` (рядом с папкой `scripts/`). Других источников секретов нет.

Стандартный набор переменных:

| Переменная | Назначение |
|---|---|
| `TRACKER_API_KEY` | API-токен / personal access token |
| `TRACKER_BASE_URL` | Базовый URL инстанса (например `https://yougile.com/api-v2`) |
| `TRACKER_PROJECT_ID` | ID проекта в трекере |
| `TRACKER_BOARD_ID` | ID доски (если применимо) |
| `TRACKER_BOT_LOGIN` | Логин, от чьего имени работает бот |

Конкретный трекер может дополнять этот список своими переменными (например `JIRA_EMAIL`, `GITHUB_OWNER`). Префикс для доп. переменных — название трекера или `TRACKER_`.

### Формат вывода

- **stdout** — всегда валидный JSON (одна строка или pretty-print — не принципиально). Никаких комментариев, логов, баннеров в stdout.
- **stderr** — человекочитаемые сообщения об ошибках, подсказки, диагностика.
- **Код возврата** — `0` при успехе, ненулевой при ошибке (`1` — общая ошибка, `2` — ошибка конфигурации/креденшиалов, `3` — ресурс не найден, `4` — сетевая/API ошибка).

### Формат ошибок

При ошибке в stdout допустимо выдать:
```json
{"error": "краткое описание", "detail": "дополнительные детали"}
```
и вернуть ненулевой код. Детали (`detail`) опциональны.

### Обработка ID

- **Идентификаторы задач** — в ответах возвращаются два поля:
  - `id` — внутренний ID трекера (UUID / числовой ID)
  - `number` — человекочитаемый номер (например `WF-21`, `#42`). Если трекер не различает — возвращай одно и то же значение в обоих полях.
- **Колонки/статусы** — в аргументах принимай **название** (человекочитаемое), скрипт сам мапит его в ID.

### Общие опции

У всех команд поддерживается:
- `--help` / `-h` — короткая подсказка в stderr, код `0`
- `--json` — по умолчанию уже JSON, флаг можно игнорировать (нужен для совместимости если когда-нибудь добавится `--text`)

## Команды

### `tracker whoami`

Вернуть информацию о пользователе, от чьего имени работает токен.

**Вывод:**
```json
{
  "id": "user-id",
  "login": "carono-ai",
  "name": "Carono AI",
  "email": "optional@example.com"
}
```

Используется агентами для самоидентификации и как smoke-test (`whoami` проходит → токен валиден).

---

### `tracker list-columns`

Вернуть список колонок/статусов на доске.

**Вывод:**
```json
[
  {"id": "col-id-1", "name": "Задачи", "position": 0},
  {"id": "col-id-2", "name": "Обсуждение", "position": 1},
  {"id": "col-id-3", "name": "Разработка", "position": 2}
]
```

Для трекеров без понятия «колонка» (Jira со статусами, GitHub Issues с лейблами) — вернуть список статусов/лейблов в том же формате. `position` — порядок в жизненном цикле (0 = самый ранний).

---

### `tracker list-tasks --column <название>`

Вернуть задачи в указанной колонке/статусе.

**Опции:**
- `--column <name>` — обязательно, имя колонки (как в `list-columns`)
- `--assignee <login>` — фильтр по исполнителю (опционально); спец. значение `none` — без исполнителя; `me` — текущий пользователь
- `--limit <N>` — макс. число задач (по умолчанию 50)

**Вывод:**
```json
[
  {
    "id": "task-id",
    "number": "WF-21",
    "title": "Заголовок задачи",
    "column": "Обсуждение",
    "assignees": ["carono-ai"],
    "url": "https://tracker/link/to/task",
    "updated_at": "2026-04-24T10:00:00Z"
  }
]
```

Пустой результат — `[]`, не ошибка.

---

### `tracker get-task <id>`

Прочитать задачу по ID (`id` из `list-tasks`, не `number`).

**Вывод:**
```json
{
  "id": "task-id",
  "number": "WF-21",
  "title": "Заголовок",
  "description": "Описание в markdown",
  "column": "Обсуждение",
  "assignees": ["carono-ai"],
  "url": "https://tracker/link",
  "created_at": "2026-04-20T10:00:00Z",
  "updated_at": "2026-04-24T10:00:00Z"
}
```

Если задача не найдена — ошибка с кодом `3`.

---

### `tracker list-comments <id>`

История комментариев к задаче в хронологическом порядке (старые первыми).

**Опции:**
- `--since <id>` — вернуть только комментарии после указанного ID (опционально, для инкрементального опроса)
- `--limit <N>` — по умолчанию 100

**Вывод:**
```json
[
  {
    "id": "comment-id",
    "author": "user-login",
    "text": "Текст комментария",
    "created_at": "2026-04-24T10:05:00Z"
  }
]
```

---

### `tracker comment <id> <text>`

Опубликовать комментарий в задачу.

**Аргументы:**
- `<id>` — ID задачи
- `<text>` — текст комментария (многострочный допустим, если shell экранирует корректно; для надёжности поддержать `--file <path>` как альтернативу)

**Опции:**
- `--file <path>` — прочитать текст из файла вместо позиционного аргумента

**Вывод:**
```json
{"id": "comment-id", "created_at": "2026-04-24T10:05:00Z"}
```

---

### `tracker move-task <id> --column <название>`

Перенести задачу в другую колонку / сменить статус.

**Вывод:**
```json
{"ok": true, "id": "task-id", "column": "Разработка"}
```

---

### `tracker assign-task <id> --user <login>`

Назначить задачу на пользователя. Спец. значения:
- `me` — текущий пользователь (владелец токена)
- `none` — снять назначение

**Вывод:**
```json
{"ok": true, "id": "task-id", "assignees": ["carono-ai"]}
```

`assignees` — полный итоговый список после операции.

---

### `tracker create-task --title <title>`

Создать новую задачу.

**Опции:**
- `--title <t>` — обязательно
- `--column <name>` — в какую колонку класть (если пропущено — в первую / дефолтную)
- `--description <d>` — описание (также поддерживать `--description-file <path>`)
- `--assignee <login>` — назначить (можно повторять флаг для нескольких)

**Вывод:**
```json
{
  "id": "new-task-id",
  "number": "WF-42",
  "url": "https://tracker/link"
}
```

---

### `tracker delete-task <id>`

Удалить (или архивировать, если удаление недоступно) задачу.

**Вывод:**
```json
{"ok": true, "id": "task-id"}
```

---

### `tracker list-users`

Список участников проекта.

**Вывод:**
```json
[
  {"id": "user-id", "login": "carono-ai", "name": "Carono AI", "email": "..."}
]
```

## Недоступные операции

Если в конкретном трекере команда физически не имеет смысла (например `list-columns` в трекере без колонок, `delete-task` если API не умеет удалять):

1. **Лучший вариант** — реализуй через близкий аналог (колонки → статусы/лейблы; удаление → архивирование/закрытие с лейблом).
2. **Если аналога нет** — возвращай:
   ```json
   {"error": "not supported", "detail": "delete-task is not supported by this tracker"}
   ```
   с кодом возврата `1`. Но такие случаи должны быть редкими — скилл должен искать обход.

---

## Требования к скрипту

- **Строгое соответствие контракту.** Имена команд, опций, формат вывода — без вариаций. Агенты парсят JSON по этому контракту.
- **Валидация входа.** Проверяй наличие обязательных аргументов, выдавай понятную ошибку в stderr.
- **Нет интерактивного ввода.** Скрипт всегда неинтерактивный — агент запускает его из bash и читает stdout.
- **Не логируй секреты.** Токен никогда не появляется в stdout/stderr, даже при ошибках.
- **Минимум зависимостей.** Bash-вариант — только `curl` + `jq`. Python-вариант — только стандартная библиотека + `requests` (если без него никак).
- **Идемпотентность где возможно.** Повторный `move-task` в ту же колонку — не ошибка, возвращает `{"ok": true}`.
