# Самопредложение улучшений плагина

Концепция работы агента-наблюдателя, который замечает шаблоны ручных вмешательств пользователя и периодически предлагает улучшения плагина. Соответствует направлению 5 ROADMAP carono-ai и принципу MISSION «человек направляет, агент исполняет»: система фиксирует сигналы и предлагает, не решает за пользователя.

Документ собирается из четырёх независимых разделов:

- `## Источник` — какой журнал служит сырьём, какие в нём поля, где проходит граница «трение vs направление». **Готов** (см. ниже, задача WF-36).
- `## Анализатор` — кто читает журнал, когда запускается, как извлекает повторяющиеся проблемы. **Готов** (см. ниже, задача WF-37).
- `## Формат вывода` — как агент-наблюдатель формулирует предложение пользователю, что включает, как пользователь принимает или отклоняет. **TODO**: задача WF-38.
- `## Проверка концепции на MISSION` — итоговая сборка и сверка с принципами и анти-целями миссии. **TODO**: задача WF-39.

## Источник

### Что считается «трением»

«Трение» — повторяющиеся ручные правки пользователя в работе агентов: одно и то же вмешательство наступает на одни и те же грабли в разных задачах или в разных запусках. Одноразовая правка трением не считается — это шум.

Граница «трение vs направление» проходит по факту попадания в журнал, а не задним числом:

- **Трение** — пользователь правит работу агента, потому что агент сделал не так. Это ошибка агента или дефект инструкции, по которой агент работает. Только такие факты попадают в журнал ручных вмешательств.
- **Направление** — пользователь меняет постановку задачи или формулирует новое требование. Не правка работы, а новый ввод. В журнал не пишется.

Решение «трение это или направление» принимает агент **в момент записи**, по правилам, уже зафиксированным в `workflow/{bot-name}/MANUAL_INTERVENTIONS.md` (раздел «Что считается вмешательством»). По построению: всё, что попало в журнал — потенциальное трение. Анализатор различает «потенциальное» от «подтверждённого» по повторяемости.

### Какой журнал — сырьё

Источник один — журнал ручных вмешательств `workflow/{bot-name}/manual-interventions.jsonl`. Контракт формата зафиксирован в шаблоне `src/templates/MANUAL_INTERVENTIONS.md` (задача WF-32) и копируется в `workflow/{bot-name}/MANUAL_INTERVENTIONS.md` при настройке бота скиллом `configure`.

Параллельного журнала «трения» не существует — это сознательное решение, чтобы:

- не плодить два потока с разной семантикой и риском рассинхронизации;
- удержать стабильный контракт записи (анализатор и пороги можно тюнить, не трогая агентов);
- сохранить принцип MISSION «контракт важнее реализации».

### Поля записи и опциональный признак трения

Обязательные поля записи (`timestamp`, `agent`, `task`, `step`, `what`) описаны в шаблоне `MANUAL_INTERVENTIONS.md`. Они одинаковые для любого вмешательства — анализатор не зависит от того, посчитал ли агент конкретный факт повторяющимся.

Поверх обязательных полей шаблон допускает два **опциональных**:

| Поле | Когда выставляется |
|---|---|
| `friction_signal: true` | Агент в момент записи субъективно уверен, что наступает на повтор (вспомнил предыдущую похожую правку, увидел тот же паттерн). По умолчанию отсутствует. |
| `severity: "minor" \| "major"` | Только если выставлен `friction_signal`. `minor` — мелкая правка (опечатка, формулировка); `major` — переделка логики или возврат к шагу. |

Это подсказка анализатору, не приговор. Окончательное решение «это трение или нет» анализатор принимает по объективным порогам (см. ниже) — наличие или отсутствие `friction_signal` на отдельной записи ничего не предрешает.

### Кто и когда определяет «это трение»

Разделение ролей:

- **Агент пишет факты.** В момент сессии, если выполнены условия из шаблона `MANUAL_INTERVENTIONS.md` («что считается вмешательством»). Сам по себе не считает повторяемость и не читает историю журнала — пишет один факт за одну идентифицированную правку. Может опционально выставить `friction_signal` если уверен.
- **Анализатор читает поток.** Это отдельный процесс (тема WF-37). Он применяет порог повторяемости поверх сырых записей и решает, какие группы записей образуют «трение» — то есть кандидата на улучшение плагина.

Такое разделение даёт:

- **стабильный контракт записи** — поля и условия записи не зависят от того, как именно анализатор считает пороги;
- **возможность тюнить пороги** под проект без правки агентов;
- **минимальную нагрузку на агента** — он не обязан читать историю журнала на каждой записи.

### Пороги повторяемости (значения по умолчанию)

Анализатор применяет следующие пороги по умолчанию (точная реализация — задача WF-37):

| Параметр | Значение по умолчанию | Что означает |
|---|---|---|
| `repeat_count` | `2` | Сколько раз должна повториться похожая запись, чтобы считаться трением. 2 повтора уже повод предлагать улучшение. |
| `window_tasks` | `10` | По скольким последним задачам считается повторяемость. Записи старше окна не учитываются. |

Пороги — **параметры анализатора**, а не часть формата записи. Каждый бот может переопределить их под себя в конфигурации (конкретное место хранения и формат описывает WF-37). При отсутствии переопределения используются значения выше.

«Похожая запись» — это группа записей по совпадению `agent + step + смысловой кластер what`. Алгоритм кластеризации `what` — деталь анализатора (WF-37), не источника.

### Чего источник намеренно не делает

- **Не классифицирует трение по типам** (формат сообщения, неверная команда, потерянный шаг и т.п.). Классификация — задача анализатора и формата вывода.
- **Не хранит severity для всего журнала.** Только опционально для записей с `friction_signal=true`. Принцип «лишние данные не пишем» из WF-32 сохраняется для общего потока.
- **Не отделяет «свежее трение» от «старого».** Окно наблюдения применяет анализатор; журнал хранит всю историю.
- **Не решает, что предлагать пользователю.** Это задача формата вывода (WF-38).

## Анализатор

### Кто читает журнал

Анализ — режим существующего скилла `analyst`. Отдельный скилл/агент не заводится: `analyst` уже умеет работать с трекером, MISSION, ROADMAP и историей прогонов, журнал ручных вмешательств для него — ещё один источник сырья. Соответствует принципу MISSION «цикл, а не инструмент»: новая способность встраивается в существующего участника, а не плодит отдельную утилиту.

Активируется отдельным режимом запуска (рабочее имя — `--mode improver`); поведение по умолчанию — `top-down` / `drill-down`, как сейчас. В режиме improver `analyst` не предлагает новые задачи из ROADMAP — только разбирает накопленный журнал и сообщает результат пользователю.

### Когда запускается

Триггер — ручной, через `check.sh` с дополнительным параметром (точное имя — деталь скилла `checker-setup`, например `./check.sh --improver` или `./check.sh --run analyst --mode improver`). Расписание и пороги «N записей в журнале» в этой итерации **не закладываются**: пока не накопится опыт реальных запусков, автоматический триггер преждевременен. Условия пересмотра — после нескольких ручных прогонов станет видно, имеет ли смысл периодичность.

### Входы

`analyst` в режиме improver читает:

| Источник | Что берётся | Зачем |
|---|---|---|
| `workflow/{bot-name}/manual-interventions.jsonl` | весь поток записей | сырьё для анализа повторяемости |
| `workflow/{bot-name}/MISSION.md` (или корневой) | принципы и анти-цели | фильтр: предлагаемое улучшение не должно противоречить миссии |
| `workflow/{bot-name}/ROADMAP.md` | текущая фаза, направления | контекст: к какому направлению приклеивается находка |
| `workflow/{bot-name}/analyst/history.md` | прошлые прогоны (включая прогоны improver) | антидубль: не предлагать то, что уже отклонено или принято в прошлый раз |

Дополнительно при необходимости — `get-task` / `list-comments` по задачам, упомянутым в записях журнала, чтобы уточнить контекст (например прочитать комментарий MR, на который ссылается запись).

### Что анализатор делает

Поведение в одном прогоне:

1. **Загружает журнал.** Читает `manual-interventions.jsonl` целиком, парсит построчно. Битые строки пропускает с предупреждением, но не падает.
2. **Фильтрует по окну наблюдения.** По умолчанию — последние `window_tasks = 10` уникальных задач (`task` в записи). Параметр окна берётся из конфигурации бота, при её отсутствии — значение из раздела «Источник».
3. **Группирует записи.** Ключ группировки — совпадение `agent + step + смысловой кластер what`. «Смысловой кластер» — это решение конкретного запуска (не алгоритм с фиксированным порогом схожести): `analyst` читает `what` целиком и собирает группы вручную, опираясь на смысл, а не на строковое равенство. Опциональный `friction_signal: true` повышает вес записи в группе, но не создаёт группы сам по себе.
4. **Применяет порог повторяемости.** Группы с `count >= repeat_count` (по умолчанию `2`) — кандидаты в «подтверждённое трение». Группы с `count < repeat_count` — пока шум, не выходят из анализатора.
5. **Сверяет с историей.** По каждой группе-кандидату проверяет `analyst/history.md`: не было ли предложение по этому же кластеру уже создано/отклонено. Если было создано — упоминает существующую задачу. Если отклонено — молчит (правило аналитика: не предлагать повторно отклонённое).
6. **Сверяет с миссией.** Если предлагаемое улучшение явно противоречит принципу или анти-цели MISSION — поднимает это отдельно (по аналогии с шагом 9.5 текущего скилла), не подмешивая в общий список.
7. **Сообщает результат пользователю в текущей сессии.** Формат — задача WF-38; здесь анализатор только готовит структурированное содержимое: список подтверждённых трений с привязкой к записям журнала и направлению ROADMAP.

### Выходы

В этой итерации — **только информирование пользователя в сессии запуска**. Анализатор:

- **не создаёт задач в трекере автоматически** — даже если кандидат прошёл все фильтры. Решение «оформлять ли это задачей» принимает пользователь по итогам сессии, обычным запуском `analyst` без режима improver или ручным созданием.
- **не пишет PR с патчем** — патчи к самому плагину остаются ручной работой; анализатор только показывает место, где плагин трётся о реальность.
- **не комментирует задачи в трекере** — это вмешательство в чужие задачи без явного запроса.

Информирование оставлено как минимальный жизнеспособный выход: пользователь увидит результат, оценит полезность, дальше решим — добавлять автоматизацию (создание задач, PR, ROADMAP-обновления) или оставить как есть.

### Запись прогона в history

Каждый запуск improver-режима добавляет запись в `workflow/{bot-name}/analyst/history.md` по аналогии с обычным прогоном `analyst`, чтобы:

- следующий прогон знал, какие группы трения уже показывались пользователю (антидубль);
- была видна история «что плагин научился ловить» — даже если в задачи это пока не оформлено.

Минимальный набор полей записи прогона improver:

- дата (абсолютная, `YYYY-MM-DD`);
- режим: `improver`;
- размер окна и порог, фактически применённые в прогоне;
- количество записей в журнале, подпавших под окно;
- список подтверждённых групп трения (короткое имя кластера + `count` + список `task`);
- решение пользователя по каждой группе: показано / отложено / отклонено / превратилось в задачу `#…`.

### Чего анализатор намеренно не делает

- **Не редактирует журнал.** `manual-interventions.jsonl` — append-only по построению WF-32; анализатор только читает.
- **Не правит MISSION/ROADMAP.** Если найденное трение указывает на несовпадение плагина с миссией — это повод для диалога, не для автоправки.
- **Не запускает других агентов.** Не вызывает `worker` / `discussion` / `reviewer` сам — только показывает находки.
- **Не классифицирует трение по типам в фиксированной таксономии** (формат, команда, шаг и т.п.). Кластеризацию делает по смыслу `what`, без принуждения к фиксированному списку категорий — это задача формата вывода (WF-38), если вообще понадобится.
- **Не привязан к конкретному трекеру.** Все обращения к задачам — через `./workflow/{bot-name}/scripts/tracker`, как и у обычного `analyst`.
