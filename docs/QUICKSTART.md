# Быстрый старт

Этот документ — последовательность из пяти шагов, чтобы поднять `carono-wf` на новом проекте и убедиться, что цикл задач работает. Рассчитан на одного разработчика, который ставит плагин впервые. Базовые знания Claude Code (`/plugin`, `/skills`, `/agents`) предполагаются — что это за плагин, объясним ниже.

---

## Что это за плагин

`carono-wf` — набор универсальных subagent'ов и скиллов для Claude Code, которые автоматизируют стандартный цикл разработки: уточнение требований → реализация → код-ревью. Агенты не привязаны к конкретному трекеру (YouGile, GitHub, GitLab, Jira и т.п.) — специфика проекта описывается в документах `workflow/{bot-name}/*.md`, которые агенты читают при запуске. Подключаешь, отвечаешь на вопросы скилла настройки — дальше агенты ходят в твой трекер сами.

---

## Предусловия

- **Claude Code** установлен и настроен.
- **git** установлен, репозиторий проекта инициализирован.

Всё остальное (токены трекера, `gh`, MCP-серверы, дополнительные права) — спросит соответствующий скилл по ходу настройки. Не нужно готовить ничего заранее.

---

## Шаг 1. Установка плагина

В Claude Code:

```
/plugin marketplace add carono/workflow
/plugin install carono-wf@carono
```

После установки в `/skills` появятся команды с префиксом `carono-wf:` (`carono-wf:configure`, `carono-wf:tracker-setup`, `carono-wf:checker-setup`, …), а в `/agents` — `carono-wf:discussion`, `carono-wf:worker`, `carono-wf:reviewer`.

---

## Шаг 2. `/carono-wf:configure`

Запусти скилл:

```
/carono-wf:configure
```

Скилл задаст вопросы про твой воркфлоу — какой трекер, какие колонки, как называется бот, какой формат коммитов и веток. Отвечай по своему процессу: правильных ответов «вообще» нет, важно зафиксировать как принято в проекте.

**Что появится после:**

```
workflow/{bot-name}/
  WORKFLOW.md     — как агенты работают с трекером и git
  PROJECT.md      — бизнес-контекст проекта (модули, интеграции, термины)
  TECH.md         — стек, паттерны, правила разработки
  bot.local.md    — учётка бота и его зоны ответственности (НЕ коммитится)
```

Имя `{bot-name}` ты задаёшь сам — обычно «технический логин» бота в трекере (например `carono-ai`, `dev-bot`).

---

## Шаг 3. `/carono-wf:tracker-setup`

```
/carono-wf:tracker-setup
```

Скилл соберёт креденшиалы трекера и сгенерирует CLI-скрипт, через который все агенты ходят в трекер. Спросит:

- какой сервис трекера (YouGile, GitHub Issues, GitLab Issues, Jira, ...)
- URL инстанса
- проект / репозиторий
- API-токен

Токен запишется в `workflow/{bot-name}/.env` (этот файл по умолчанию в `.gitignore`). Дальше скилл сделает живые пробы API трекера, посмотрит структуру ответов и сгенерирует скрипт под фактические имена полей.

**Что появится после:**

```
workflow/{bot-name}/
  .env                     — токен и креденшиалы (не коммитится)
  scripts/tracker          — CLI-скрипт, общий для всех трекеров
```

В конце скилл прогонит smoke-test: создаст тестовую задачу, проверит все 11 команд (`whoami`, `list-tasks`, `get-task`, `comment`, `move-task`, ...), удалит тестовую задачу. Если smoke-test зелёный — интеграция работает.

---

## Шаг 4. `/carono-wf:checker-setup`

```
/carono-wf:checker-setup
```

Скилл прочитает `bot.local.md` (зоны ответственности агентов) и сгенерирует `scripts/check.sh` — скрипт, который смотрит состояние задач в твоей доске и запускает нужного агента (`discussion` / `worker` / `reviewer`) под подходящую задачу.

**Что появится после:**

```
workflow/{bot-name}/
  scripts/check.sh         — проверка состояния задач + запуск агентов
```

Запускать руками: `./workflow/{bot-name}/scripts/check.sh` — увидишь сводку по колонкам. С флагом `--run` — поднимет агента под первую подходящую задачу.

---

## Шаг 5. Проверка воркфлоу на реальной задаче

Возьми **любую короткую задачу** из своего бэклога. Если бэклог пустой — заведи маленькую руками (добавить раздел в README, мелкий рефакторинг одного модуля, поправить опечатку в документации). Главное чтобы:

- задача формулировалась одной-двумя фразами,
- решалась за 10–30 минут чистого времени агента,
- не требовала радикальных решений по архитектуре.

Перенеси задачу в колонку, которую `bot.local.md` указывает как стартовую для `discussion` (обычно «Обсуждение» или эквивалент). Назначь её на бота или оставь без исполнителя — `bot.local.md` определяет, что бот возьмёт.

### Прогон через discussion → worker

1. **Запусти агента discussion:**
   ```
   /carono-wf:discussion
   ```
   Агент возьмёт задачу, прочитает описание и комментарии, при необходимости задаст уточняющие вопросы, после ответов — соберёт summary. **Перевод задачи в «Разработку» делает человек** — это намеренно: агент не решает за тебя, что задача готова.

2. **Когда summary тебя устраивает** — переведи задачу в колонку «Разработка» руками (или через `./scripts/tracker move-task <id> --column "Разработка"`).

3. **Запусти агента worker:**
   ```
   /carono-wf:worker
   ```
   Агент возьмёт задачу из «Разработки», создаст ветку, реализует, откроет MR/PR, перенесёт задачу в «На проверке».

Если оба шага прошли без вмешательства — воркфлоу настроен. `reviewer` в этом quickstart намеренно не запускаем: он работает на следующем витке, когда задач уже несколько и ревью идёт параллельно.

## Что дальше

- **`README.md`** — общий обзор плагина, ссылки на исходники.
- **`workflow/{bot-name}/WORKFLOW.md`, `PROJECT.md`, `TECH.md`** — твой воркфлоу. Дополняй их по мере того, как агенты обнаруживают новые особенности проекта (они умеют сами дописывать `TECH.md` / `knowledge/` после задачи).
- **Остальные скиллы** — `/carono-wf:tracker-check` (диагностика интеграции если что-то отвалилось), `/carono-wf:analyst` (нарезка задач из `MISSION.md` / `ROADMAP.md`).

Если плагин ставится не в первый раз и интеграция вдруг перестала работать — сразу `/carono-wf:tracker-check`, не возись руками в `scripts/tracker`.
