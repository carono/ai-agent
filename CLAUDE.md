# ai-agent

Мы разрабатываем **набор универсальных агентов для автоматизации разработки** — переиспользуемый шаблон, который подключается к любому проекту.

## Суть

Агенты автоматизируют стандартный цикл разработки (обсуждение → разработка → код-ревью), адаптируясь под воркфлоу конкретного проекта через `docs/WORKFLOW.md`. Не привязаны к конкретному трекеру (GitLab, GitHub, Jira и т.д.) — вся специфика описана в документах проекта, которые агенты читают при запуске.

## Агенты

- **configure** — одноразовый: расспрашивает пользователя о воркфлоу и создаёт `workflow/{bot-name}/WORKFLOW.md`, `PROJECT.md`, `TECH.md`, `bot.local.md`
- **discussion** — уточняет требования по задачам, переводит в "готово к разработке"
- **worker** — реализует задачи, создаёт код-ревью
- **reviewer** — разбирает замечания к коду, закрывает задачи
- **doc (agents-doc-updater)** — актуализирует человекочитаемую документацию по агентам
- **tracker** — диагностика интеграции с трекером

## Скиллы

- **tracker-setup** — задаёт вопросы про трекер, собирает креденшиалы в `workflow/{bot-name}/.env`, **через живые пробы API** (не по шаблонам) генерирует CLI-скрипт `workflow/{bot-name}/scripts/tracker` — единую точку входа к трекеру для всех агентов. Перед записью каждой команды делает `curl` к реальному эндпоинту, пишет маппинг под фактические имена полей.
- **checker-setup** — читает `bot.local.md`, генерирует `workflow/{bot-name}/check.sh` — скрипт проверки состояния задач и запуска агентов (опирается на CLI `scripts/tracker`)

## Агент или скилл?

- **Агент** — обработка задач в трекере. Работает по зоне ответственности из `bot.local.md` (колонка + фильтр), запускается из `check.sh --run` или по требованию, завершается когда в его колонке нет подходящих задач.
- **Скилл** — разовая генерация артефакта (`.env`, `scripts/tracker`, `check.sh`, …) через диалог с пользователем. Запускается по нужде — первичная настройка, ротация токена, изменение зон ответственности — и завершается сразу после создания файла.

Правило выбора: делает артефакт и завершается → скилл. Обрабатывает поток задач → агент.

## Распространение

Репозиторий `carono/workflow` — одновременно маркетплейс `carono` и плагин `carono-wf`. Установка в Claude Code:
```
/plugin marketplace add carono/workflow
/plugin install carono-wf@carono
```

В Claude Code все компоненты получают префикс `carono-wf:` автоматически (`/carono-wf:tracker-setup`, `carono-wf:worker` и т.д.).

Для OpenCode плагин не работает — там ставят через `npx degit carono/workflow/dist/opencode/...` (см. README).

## Принцип

Агенты не хардкодят названия статусов, API, инструменты. Всё читается из `workflow/{bot-name}/*.md` конкретного проекта в рантайме.

Взаимодействие с трекером — **только через CLI-контракт** `workflow/{bot-name}/scripts/tracker`. Контракт единый для любого трекера (YouGile, GitHub, GitLab, Jira, …), реализация — отдельный скрипт под конкретный API. Прямых HTTP/MCP-вызовов в агентах быть не должно: когда API трекера меняется, агенты об этом не узнают — меняется только скрипт (перегенерацией `tracker-setup`).

## Правило разработки

Всё что разрабатывается для агентов — файлы, логи, стейт, конфиги, скрипты — должно быть привязано к рабочей папке бота (`workflow/{bot-name}/`). Никакие артефакты не должны создаваться в корне репозитория или в общих директориях вне контекста конкретного бота.

## Сборка и шаблонизация

- **Исходники** — `src/`
- **Собранные артефакты:**
  - `agents/` и `skills/` в **корне репо** — структура плагина Claude Code (`carono-wf`)
  - `dist/opencode/` — для OpenCode-пользователей, ставится через degit
- **Манифесты плагина** — `.claude-plugin/plugin.json` и `.claude-plugin/marketplace.json`, статические, не генерируются сборкой
- **Правило**: правки вносятся ТОЛЬКО в `src/`. Файлы в `agents/`, `skills/`, `dist/opencode/` пересобираются из `src/` и руками не редактируются (хотя коммитятся в репозиторий — пользователи ставят из git).

### Соответствие src → артефакт

| Исходник | В плагине (корень репо) | В OpenCode (`dist/opencode/`) |
|---|---|---|
| `src/agents/*.md` | `agents/*.md` | `agents/*.md` |
| `src/tools/*.md` | `agents/*.md` (слиты вместе) | `tools/*.md` (отдельно) |
| `src/skills/<name>/` | `skills/<name>/` | `skills/<name>/` |

Плагин Claude Code экспонирует только одну папку subagent'ов — `agents/`. Разделение `src/agents/` vs `src/tools/` в исходниках логическое (бизнес-агенты vs сервисные), не структурное.

### Шаблонизация файлов платформы

Когда в исходниках агентов/скиллов (`src/agents/`, `src/tools/`, `src/skills/*/SKILL.md`) нужно сослаться на файл из `src/rules/` или `src/templates/`, ссылка заменяется на шаблонную переменную вида `{{VARIABLE}}`. При сборке `build.py` подставляет содержимое файла вместо переменной.

**Правило именования переменных:**

| Папка источника | Формат переменной | Пример |
|---|---|---|
| `src/rules/<имя>.md` | `{{RULES_<ИМЯ>}}` | `src/rules/security.md` → `{{RULES_SECURITY}}` |
| `src/templates/<имя>.md` | `{{TEMPLATE_<ИМЯ>}}` | `src/templates/WORKFLOW.md` → `{{TEMPLATE_WORKFLOW}}` |

**Пример:** вместо «Прочитай `rules/security.md` и выполни инструкции» пишем `{{RULES_SECURITY}}`.

Если добавляешь новый файл в `rules/` или `templates/` или другие, который должен быть встроен в агентов — добавь соответствующую переменную в `build.py::load_template_variables()`.

## Коммит

**Обязательное правило перед коммитом:**

1. **Собери проект** — запусти `python3 src/scripts/build.py` и убедись что сборка прошла без ошибок
2. **Проверь артефакты на синтаксические ошибки** — валидный JSON у манифестов, корректный YAML frontmatter, рабочий bash/python код
3. **Только после успешной сборки и проверки** — создавай коммит

Никогда не коммить изменения без предварительной сборки. Артефакты в `agents/`, `skills/`, `dist/opencode/` должны соответствовать актуальному состоянию `src/`.
