# MISSION

## Зачем это существует

Автоматизировать стандартный цикл разработки так, чтобы он работал в любом проекте без доработки агентов. Разработчик подключает плагин, описывает свой воркфлоу — и агенты сразу начинают работать.

---

## Принципы, по которым принимаются решения

**Универсальность важнее удобства.** Любая специфика конкретного проекта (трекер, статусы, API, соглашения) должна жить в документах проекта, а не в коде агентов. Если что-то нельзя вынести в документы — это архитектурная проблема.

**Агент — читатель, не исполнитель инструкций хардкода.** Агент получает задачу, читает контекст из документов, принимает решение. Не следует жёстко заданным шагам — следует принципам.

**Минимальный барьер входа.** Одна команда подключения, без настройки, без конфигов. Если для запуска нужно сделать больше одного шага — это регресс.

**Цикл, а не инструмент.** Ценность не в отдельном агенте, а в замкнутом цикле: обсуждение → разработка → ревью → закрытие. Каждый агент — звено, а не самостоятельная утилита.

**Документы живут с кодом.** Агенты дополняют проектную документацию по мере работы. Знания о проекте накапливаются автоматически, не требуя отдельных усилий команды.

**Контракт важнее реализации.** Интеграция с внешней системой описывается коротким стабильным контрактом — точкой, от которой работают все агенты. Реализация контракта собирается под актуальный API: через живые пробы, не по шаблонам и не по памяти. Шаблоны устаревают тихо; контракт стабилен и меняется редко.

---

## Что НЕ является целью

- Создать агентов под конкретный трекер или стек
- Покрыть все возможные воркфлоу из коробки
- Заменить человека в принятии решений — агенты исполняют, человек направляет

---

## Для агентов разработки

При принятии архитектурных решений, добавлении новых агентов или изменении существующих — сверяйся с этими принципами. Вопросы-маркеры:

- Это работает в любом проекте или только в конкретном? → если только в конкретном, вынести в документацию проекта
- Это звено цикла или изолированный инструмент? → звенья приоритетны
- Усложнит ли это подключение плагина? → если да, искать более простое решение
- Это фиксируется как контракт или как код-шаблон? → код устареет, контракт переживёт смену API
