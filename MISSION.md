# MISSION

## Зачем это существует

Дать проекту способность жить самостоятельно. После подключения и настройки цикл — исследование, постановка задач, обсуждение, разработка, опциональное тестирование, коммит, переосмысление своего пути — продолжается без участия человека. Человек подключается когда хочет: внутри цикла как ещё один агент или над циклом, задавая направление. Его отсутствие не останавливает проект; его присутствие не требуется, чтобы цикл двигался.

---

## Принципы, по которым принимаются решения

**Автономия по умолчанию.** Цикл должен катиться без человека. Любая фича, требующая его обязательного участия для прогресса — регресс. Спрашивать только когда вмешательство объективно необходимо: ситуация выходит за рамки выводимого из контекста, и цена ошибки невосстановима средствами самого цикла.

**Универсальность важнее удобства.** Любая специфика конкретного проекта (трекер, статусы, API, соглашения) должна жить в документах проекта, а не в коде агентов. Если что-то нельзя вынести в документы — это архитектурная проблема.

**Агент — читатель, не исполнитель хардкода.** Агент получает задачу, читает контекст из документов, принимает решение. Не следует жёстко заданным шагам — следует принципам. Шаги в промпте — каркас, не догма.

**Минимальный барьер входа.** Одна команда подключения, без настройки, без конфигов. После подключения система живёт сама — никакого ежедневного диспетчирования или согласования шагов.

**Замкнутый цикл с саморефлексией.** Контур: исследование → постановка → обсуждение → разработка → тестирование (опционально) → коммит → переосмысление. ROADMAP проект корректирует сам по результатам цикла. MISSION — только человек: это якорь, который не должен дрейфовать без него.

**Документы живут с кодом.** Агенты дополняют проектную документацию по мере работы. Знания о проекте накапливаются автоматически, как побочный продукт цикла, а не как отдельная нагрузка на команду.

**Контракт важнее реализации.** Интеграция с внешней системой описывается коротким стабильным контрактом — точкой, от которой работают все агенты. Реализация контракта собирается под актуальный API через живые пробы, не по шаблонам и не по памяти. Шаблоны устаревают тихо; контракт стабилен и меняется редко.

---

## Что НЕ является целью

- Создать агентов под конкретный трекер или стек
- Покрыть все возможные воркфлоу из коробки
- Требовать присутствия человека для прогресса цикла — он опциональный участник, не диспетчер
- Позволять агентам самостоятельно менять собственную миссию — MISSION фиксируется человеком, иначе якорь теряет смысл

---

## Для агентов разработки

При принятии архитектурных решений, добавлении новых агентов или изменении существующих — сверяйся с этими принципами. Вопросы-маркеры:

- Это работает без человека или требует его участия? → если требует, это исключение, ищи способ убрать
- Это работает в любом проекте или только в конкретном? → если только в конкретном, вынести в документацию проекта
- Это звено цикла или изолированный инструмент? → звенья приоритетны
- Это меняет MISSION или ROADMAP? → ROADMAP корректируем сами, MISSION предлагаем человеку, не правим
- Усложнит ли это подключение плагина или ежедневное самостоятельное движение цикла? → если да, ищи более простое решение
- Это фиксируется как контракт или код-шаблон? → код устареет, контракт переживёт смену API
- Стоит ли остановить цикл и спросить человека? → только если последствия ошибки невосстановимы средствами цикла
